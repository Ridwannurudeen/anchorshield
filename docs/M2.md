# M2 Reusability Proof

## Scope

M2 adds the second AnchorShield gate:

- `testdata/rwa`: a second Groth16 proof generated from the same credential witness and same `eligibility.circom` setup.
- `contracts/gate_rwa`: Soroban proof gate with co-located M2 issuer root, policy, nullifier, verifier, and mock RWA holding ledger.
- A cross-gate regression test proving the payment and RWA policies accept the same credential root and produce distinct nullifiers.

Mock scope is explicit: the RWA asset is an internal mock ledger. M2 verifies the current SEP-57/OpenZeppelin RWA surface and models the gate boundary, but it does not deploy the full OpenZeppelin RWA stack.

## SEP-57 Verification

Verified source state:

- SEP-57 is draft v0.3.0, updated 2026-06-11.
- OpenZeppelin `stellar-contracts` `main` was checked locally at commit `b5065e23c8fd1fe15d70ccecaa94aa6d8f6d9611`.
- Current OpenZeppelin RWA compliance hooks are `transferred`, `created`, and `destroyed`.
- Current identity verifier functions are `verify_identity` and `recovery_target`.
- Current `RWAToken` includes `mint`, `forced_transfer`, `burn`, freeze functions, `set_compliance`, and `set_identity_verifier`.

The M2 contract is therefore described as SEP-57-shaped, not as a production RWA issuer deployment.

## Contract

Contract path: `contracts/gate_rwa`

The contract enforces:

- registered credential root equals public `credential_root`
- policy fields equal the public proof inputs
- `action_type == 1`
- real RWA action args equal the public proof inputs
- terms hash arg equals public `packet_hash`/terms hash
- nullifier is unused before transfer and marked used after transfer
- Groth16 pairing verification succeeds
- mock RWA inventory has enough balance before crediting the recipient holding

Local tests:

```bash
cd contracts/gate_rwa
cargo test -- --nocapture
```

Result: `7 passed`.

Covered paths:

- same credential satisfies payment and RWA policies
- valid proof executes mock RWA transfer
- payment proof cannot execute RWA gate
- RWA proof for action A fails for action B
- wrong RWA policy parameters reject
- terms hash mismatch rejects
- reused nullifier rejects
